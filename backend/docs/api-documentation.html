Complete reference for the European Union backend. For frontend integration. Based on BRD v2.0 (03 May 2026).

Overview & Authentication Required on every request

Base URL (dev): http://localhost:3333/api/v1

Step 1 — Login to get your token
POST /auth/sign-in Get access token

Public endpoint — no auth headers needed. Call this first to get your accessToken.

Request Body
FieldTypeRequiredExample
emailstringrequired"akhil.sachan@coditas.com"
passwordstringrequired"Admin@123"
Response 201
// Use accessToken in all subsequent requests
{
  "accessToken": "eyJhbGci...",
  "refreshToken": "eyJhbGci...",
  "roles": ["ADMIN"],
  "activeRole": "ADMIN",
  "email": "akhil.sachan@coditas.com",
  "expiresIn": "28d"

Step 2 — Required headers on ALL financial API calls
Authorization: Bearer <accessToken>
X-Active-Role: ADMIN ← current user's active role
X-Correlation-Id: <any-uuid> ← unique per request for tracing
X-Source-Type: WEB
X-Client-Type: WEB
Content-Type: application/json ← for POST/PATCH
⚠️ Missing any header → 400 Bad Request. Missing Authorization → 401 Unauthorized. Missing permission for the action → 403 Forbidden.

Document Flow BRD §3 — The chain every financial action follows

Every financial document follows a strict parent→child chain. A child cannot exist without an approved parent.

→

Invoice

→

Required first

Invoice
→
Book Payment
Multiple OK

Bank Transfer
1:1 with BP
→
Payment Advice
Auto-generated
🔑 Key constraints
  • Each step requires the previous to be APPROVED before the next can be created
  • 1 JMC = 1 Report = 1 Invoice — duplicates rejected
  • Σ Invoices per PO cannot exceed PO totalAmount
  • PURCHASE Bank Transfer amount must exactly equal Book Payment amount
  • SALE Bank Transfer allows multiple partials (Σ ≤ Invoice total)

File Upload Required before creating any financial document

All financial documents require an attachment (fileKey + fileName). Upload the file first, get back the keys, then include them in the document create body. Supports PDF, JPEG, PNG — max 10 MB per file.
POST /files/financial-upload Upload one financial document attachment
🔒 Any authenticated user

Upload a single PDF or image to S3. Returns the S3 key and original filename. Use these values in the fileKey and fileName fields of any financial document create call.

Request — multipart/form-data
FieldTypeRequiredRules
financialFileFile (binary)requiredPDF, JPEG, PNG only. Max 10 MB.
Response 201
{
  "fileKey": "financial-files/1778120857117_po-doc.pdf",
  "fileName": "po-doc.pdf"
}
Error Cases
StatusReason
400No file provided / invalid file type (e.g. .txt, .csv) / file exceeds 10 MB
401Missing or invalid authorization token
GET /files?key=<fileKey> Get presigned download URL
🔒 Any authenticated user

Exchange a stored S3 key for a time-limited presigned URL to download/preview the file in the browser.

Query Parameters
ParamTypeRequired
keystringrequired — the fileKey returned from /files/financial-upload
Response 200
{
  "url": "https://s3.amazonaws.com/bucket/financial-files/...?X-Amz-Signature=...",
  "key": "financial-files/1778120857117_po-doc.pdf"
}

Purchase Orders (POs) BRD §4.1 — Step 1 in the document chain

What is a PO? A Purchase Order establishes the maximum financial ceiling for a contractor (SALE) or vendor (PURCHASE) at a site. All subsequent invoices for that party must not exceed this PO total. Requires admin approval before a JMC can be created.
POST /purchase-orders Create a PO (SALE or PURCHASE)
🔒 financials.purchase-orders.create
Request Body
FieldTypeRequiredNotes
siteIduuidrequiredSite this PO belongs to
partyTypeSALE | PURCHASErequiredSALE=contractor pays us, PURCHASE=we pay vendor
contractorIduuidrequired if SALEMust be linked to the site
vendorIduuidrequired if PURCHASEMust be linked to the site
poNumberstringrequiredUnique within (site, partyType)
poDateISO date stringrequirede.g. "2026-04-01"
taxableAmountnumberrequiredPre-GST amount
gstAmountnumberoptionalDefault 0
totalAmountnumberrequiredMust equal taxableAmount + gstAmount exactly
fileKeystringrequiredS3 key from /files/financial-upload
fileNamestringrequiredOriginal file name
remarksstringoptional
⚠️ Business rules
  • totalAmount must exactly equal taxableAmount + gstAmount — rejected otherwise
  • Only one of contractorId/vendorId is set, must match partyType
  • Creates in PENDING state — a JMC cannot be created until this PO is APPROVED
  • Multiple POs allowed for the same contractor/vendor on the same site
Response 201
{ "message": "Purchase order created successfully", "id": "uuid" }

🔒 financials.purchase-orders.view

Returns full PO with contractor/vendor/site relations and the denormalized rollup columns: invoicedTotal, bookedTotal, paidTotal, lastInvoiceAt, lastPaymentAt.

PATCH /purchase-orders/:id Update PO (PENDING state only)
🔒 financials.purchase-orders.update
⚠️ Rules
  • Can only edit while approvalStatus = PENDING and isLocked = false
  • Cannot change siteId, partyType, contractorId, or vendorId after creation
  • If changing amounts, all three (taxable, gst, total) must remain consistent
Request Body (all fields optional)

POST /purchase-orders/:id/approve Approve a PO (Admin only)
🔒 financials.purchase-orders.approve

Request Body (optional)

Response 201

POST /purchase-orders/:id/reject Reject a PO (Admin only — PENDING only)
🔒 financials.purchase-orders.approve
Request Body
{ "reason": "Amount exceeds budget allocation" }
Guard
  • Returns 400 if the PO is already APPROVED. An approved document cannot be rejected — use the unlock flow to make changes.
POST /purchase-orders/:id/unlock-request Request admin to unlock an approved PO for editing
🔒 financials.purchase-orders.update

Use this when an approved (locked) PO needs correction. Once the admin grants the unlock, the PO returns to PENDING and can be edited, then re-approved.

Request Body
{ "reason": "Need to correct the PO amount" }
POST /purchase-orders/:id/unlock-grant Admin grants unlock — PO returns to PENDING
🔒 financials.purchase-orders.unlock

Admin-only. After this, the PO is in PENDING state, isLocked=false, and the user can edit it. Must be re-approved after editing.

POST /purchase-orders/:id/unlock-reject Admin rejects unlock request — PO stays locked
🔒 financials.purchase-orders.unlock

Dismisses the pending unlock request without changing the PO state. approvalStatus remains APPROVED, isLocked remains true. The unlock request fields (unlockRequestedAt, unlockRequestedBy, unlockReason) are cleared. Returns 400 if no unlock request is pending.

Response 200
{ "message": "Unlock request rejected — document remains locked" }
Error Responses
400 No pending unlock request found on this document.
404 Purchase order not found

JMCs — Job Measurement Certificates BRD §4.2 — Step 2

What is a JMC? A JMC certifies how much work has been completed against a specific PO. It is the foundation for all subsequent billing. Multiple JMCs can exist per PO, but each JMC leads to exactly one Invoice. Requires admin approval.

  • Parent PO must be APPROVED — 400 if not

Records the actual bank transaction. PURCHASE: Links to a Book Payment, amount must match exactly, auto-generates a Payment Advice. SALE: Links directly to Invoice, multiple partial transfers allowed (Σ ≤ invoice total). Both sides auto-approved.
